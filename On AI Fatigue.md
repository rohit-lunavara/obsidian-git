> [!QUOTE] Original Capture
> There is a big shift in the way software engineers are working right now compared to the pre-AI era
> 	How I used to work: Start working on a problem, work with intense focus on a task for a few hours, starting with TDD/paper design, keep iterating and learning about different business concepts, libraries, internal systems and their interactions, finish the task on a logical endpoint for the day, repeat next day till task is done
> 	How I work now: Start working on a problem, provide a rough prompt to an AI agent describing the problem and ask it to generate 2-3 candidate approaches and pick the one that's most appropriate, generate a plan and skim through it for inconsistencies and vague implementation guidelines, kick off the agent and work on another task or more commonly - switch to YouTube, Instagram, Discord, etc; once the milestone is implemented I review the code and ask the AI agent to continue, repeat till task is done
> The issue with the AI approach is there's very little time to deeply understand the code that is being generated, so most of the changes are being rubber stamped right now and I rely mainly on CI, QA and observability to catch issues
> 	As I've learnt in Advanced SWE class, the time to identify, resolve and deploy a fix for a bug increases exponentially the later we pick it, so, if we move bugs further right, then it will be much more costly to fix them
> Why not slow down?
> 	Personally, I find working with AI to be a pleasant experience overall since I can explore the design space much more and the drudgery of implementation is not handled by me
> 	However, if more of my time will be spent debugging which is much more unpleasant compared to implementation, I'm not sure if this is a positive change overall
> 	Maybe, I need to figure out a balanced medium approach of using AI, but still being able to deeply understand what's happening in the codebase
